# Vector Database — конспект

## 1. Что такое Vector Database

**Vector Database (Vector DB)** — база данных, оптимизированная для хранения и поиска векторов (эмбеддингов).

Главная цель:

* быстро находить ближайшие по смыслу объекты (nearest neighbors)
* поддерживать semantic search и RAG

---

## 2. Отличие от обычной БД

| Обычная БД       | Vector DB                  |
| ---------------- | -------------------------- |
| Ищет exact match | Ищет близость векторов     |
| Индексы на ключи | Индексы на embedding space |
| SQL / NoSQL      | FAISS, HNSW, Annoy, Milvus |

---

## 3. Основные компоненты

1. **Эмбеддинги** — векторы текстов или документов
2. **Индекс** — структура для быстрого поиска (Flat, HNSW, IVF)
3. **Search engine** — nearest neighbor search (cosine similarity, L2)
4. **Metadata** — оригинальный текст, ссылки, id

---

## 4. Как это работает

1. Документы → эмбеддинги
2. Эмбеддинги → индекс
3. Запрос → эмбеддинг
4. Поиск top-k ближайших → контекст для LLM

Пример визуально:

```
query embedding → [nearest embeddings in vector DB] → retrieve documents → LLM generates answer
```

---

## 5. Популярные Vector DB

* **FAISS** (Facebook) — локальный, быстрый, open-source
* **Chroma** — удобен для RAG + Python
* **Weaviate** — REST API + ML фичи
* **Milvus** — корпоративный, масштабируемый

---

## 6. Типы индексов

| Тип  | Особенности                  | Скорость | Память   |
| ---- | ---------------------------- | -------- | -------- |
| Flat | точный поиск                 | медленно | высоко   |
| HNSW | approximate nearest neighbor | быстро   | умеренно |
| IVF  | inverted file                | быстро   | экономно |

---

## 7. Метрики поиска

* **Cosine similarity** — стандарт для эмбеддингов
* **Euclidean distance (L2)** — альтернативный способ измерения близости
* **Recall@k, MRR, nDCG** — оценка качества retrieval

---

## 8. Интеграция с RAG

* Vector DB хранит **семантические эмбеддинги документов**
* LLM делает генерацию **на основе retrieved context**
* Важные параметры:

  * top-k → сколько документов подставлять
  * размер chunk → ≤ max_tokens LLM
  * нормализация векторов → для cosine similarity

---

## 9. Практические советы

* Разбивай длинные документы на чанки
* Кэшируй embeddings, чтобы ускорить поиск
* Для multilingual задач выбирай подходящий embedder
* Выбирай индекс под задачу: точность vs скорость

---

## 10. Интуитивно

Vector DB = «карта смыслов» всех документов:

* query → находим ближайшие точки → получаем контекст
* LLM получает только то, что реально нужно, а не весь текст

---

# Краткое резюме

Vector Database — ядро Retrieval-Augmented Generation:

* хранит семантические embeddings
* быстро ищет релевантные документы
* поддерживает качественные и быстрые ответы LLM
* оптимизирует RAG по скорости и точности
