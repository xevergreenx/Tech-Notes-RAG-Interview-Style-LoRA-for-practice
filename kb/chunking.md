# Chunking — конспект

## 1. Что такое Chunking

**Chunking** — это разбиение текста на небольшие части (чанки) для обработки LLM или построения embeddings.

Цель:

* уложиться в контекстное окно модели
* улучшить точность retrieval в RAG
* оптимизировать генерацию и поиск

---

## 2. Почему важен Chunking

* LLM имеет ограничение на **максимальное количество токенов** (например, 4096)

* Длинные документы нужно разбивать, чтобы:

  * каждый чанк помещался в модель
  * модель могла использовать релевантный контекст для генерации ответа

* В RAG каждый чанк → отдельный embedding → отдельная запись в vector DB

---

## 3. Подходы к Chunking

### 3.1 Fixed-size chunks

* Разбиваем текст на чанки фиксированной длины (например, 512 токенов)
* Плюсы: просто
* Минусы: можно резать смысл, делить предложения

### 3.2 Sliding window

* Наложение чанков с перекрытием (например, stride = 256 при chunk_size = 512)
* Плюсы: минимизируем потерю контекста
* Минусы: увеличивается количество эмбеддингов → больше памяти

### 3.3 Semantic / Sentence-level

* Разбиваем по предложениям или абзацам
* Плюсы: сохраняем смысл
* Минусы: чанки разной длины → иногда приходится паддить

---

## 4. Пример кода (Python + Hugging Face tokenizer)

```python
from transformers import AutoTokenizer

tokenizer = AutoTokenizer.from_pretrained("Qwen/Qwen2.5-7B-Instruct")
text = "Очень длинный текст, который нужно разбить на чанки..."

# параметры
chunk_size = 128
stride = 64

tokens = tokenizer(text)["input_ids"]

chunks = []
for i in range(0, len(tokens), stride):
    chunk = tokens[i:i+chunk_size]
    if len(chunk) == 0:
        break
    chunks.append(chunk)

# преобразуем обратно в текст (для embeddings или хранения)
chunk_texts = [tokenizer.decode(c, skip_special_tokens=True) for c in chunks]
```

---

## 5. Чанки и RAG

* Каждый чанк → embedding → vector DB
* Запрос → embedding → поиск top-k ближайших чанков
* Формируем prompt для LLM:

```
Контекст: <retrieved chunks>
Вопрос: <user query>
Ответь:
```

* Правильный chunking = релевантный контекст без переполнения окна модели

---

## 6. Практические советы

* Для длинных документов лучше sliding window
* Поддерживай баланс: размер чанка ≈ max_tokens / 2, stride ≈ chunk_size / 2
* Проверяй, чтобы чанки не обрывали смысловые блоки
* Для multilingual текста используем multilingual tokenizer

---

## 7. Интуитивно

Chunking = «нарезка текста на удобные куски», чтобы:

* модель могла обработать весь контент
* RAG точно находил релевантные части
* не переполнялось контекстное окно LLM

---

# Краткое резюме

Chunking — ключевой этап подготовки текста для LLM и RAG:

* Чанки фиксированной длины, sliding window или sentence-level
* Каждый чанк → embedding → vector DB
* Баланс между размером чанка и перекрытием → оптимальная точность retrieval
* Правильный chunking = точные и релевантные ответы модели
