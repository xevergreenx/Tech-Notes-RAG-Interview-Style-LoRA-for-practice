# RAG — конспект

## 1. Что такое RAG

**RAG (Retrieval Augmented Generation)** — это архитектура, где LLM отвечает на вопрос, используя внешнюю базу знаний, найденную через поиск по эмбеддингам.

Формула:
LLM + поиск по документам = RAG

Зачем нужен:

* уменьшает галлюцинации
* даёт актуальные знания
* позволяет работать с приватными данными

---

## 2. Общий pipeline

1. Источники данных
2. Чанкинг документов
3. Создание эмбеддингов
4. Сохранение в vector DB
5. Поиск релевантных кусков
6. Передача контекста в LLM
7. Генерация ответа

---

## 3. Компоненты системы

### 3.1 Документы

Это база знаний:

* pdf
* txt
* html
* базы данных

Главное правило:
один чанк = одна завершённая мысль

---

### 3.2 Чанкинг

Разбиение текста на куски.

Типичные параметры:

* 300–1000 токенов
* overlap 10–20%

Зачем overlap:
чтобы не терялся контекст между кусками

---

### 3.3 Эмбеддинги

Эмбеддинг = вектор, отражающий смысл текста.

Используются для:

* поиска
* сравнения

Свойства:

* фиксированная размерность
* семантическая близость → близкие вектора

---

### 3.4 Vector Database

Хранилище эмбеддингов.

Примеры:

* FAISS
* Chroma
* Weaviate

Функция:
быстрый поиск ближайших векторов (nearest neighbors)

---

### 3.5 Retrieval

Процесс поиска релевантных документов.

Шаги:

1. эмбеддинг запроса
2. поиск top-k документов
3. сбор контекста

---

### 3.6 Генерация

LLM получает:

* вопрос
* найденные документы
* инструкцию

И генерирует ответ с опорой на контекст.

---

## 4. Prompt для RAG

Хороший prompt должен:

* явно говорить использовать контекст
* запрещать выдумывать
* просить цитировать

Пример структуры:

Система:
Ты эксперт. Отвечай только на основе контекста.

Контекст:
{retrieved_docs}

Вопрос:
{query}

---

## 5. Метрики качества

### Retrieval

* Recall@k
* Precision

### Ответ

* factual accuracy
* groundedness
* latency

---

## 6. Когда нужен fine-tuning

Нужен если:

* требуется стиль
* нужен формат ответа
* узкая предметная область

Не нужен если:

* знания меняются
* важна актуальность

---

## 7. Типичные ошибки

❌ слишком большие чанки
❌ плохие эмбеддинги
❌ нет инструкций в prompt
❌ k слишком маленький
❌ отсутствие reranker

---

## 8. Улучшения RAG

* reranking моделей
* hybrid search (BM25 + embeddings)
* query rewriting
* caching
* citations

---

## 9. Минимальный стек (MVP)

LLM: 7–8B instruct
Embedder: multilingual retrieval model
Vector DB: FAISS
Backend: Python

---

## 10. Интуитивное понимание

RAG =
модель не «помнит всё»,
она **ищет перед тем как ответить**.

Это делает систему:

* точнее
* честнее
* масштабируемой

---

# Краткое резюме

RAG — это способ превратить LLM
из генератора текста в систему,
которая отвечает на основе знаний.

Главная идея:
не учить модель всему,
а давать ей нужный контекст в момент запроса.
